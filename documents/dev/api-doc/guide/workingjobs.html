7 Working with the Job APIs

The Job APIs facilitate working with DeployR managed jobs. Jobs support the execution of commands in the background on behalf of users.

Submitting Jobs v Scheduling Jobs

Jobs can be submitted for immediate background execution using the /r/job/submit call. Jobs can be scheduled for execution at some later date or on a recurring basis using the /r/job/schedule call.

Job Scheduling Priority

When a user submits or schedules a job they can identify a high, medium or low (default) priority for their job. Each priority level has it's own dedicated jobs queue managed by the jobs scheduler.

Jobs in higher priority job queues are guaranteed to be executed on the grid ahead of jobs in lower priority job queues. Put another way, jobs in higher priority job queues are assigned free slots on the grid ahead of jobs on lower priority queues. Jobs on the same jobs queue are executed on the grid in the order in which they were submitted.

High, medium and low priority job queues allow users to expedite higher priority jobs by 'jumping the queue', and to expedite jobs to best fit specific scheduling needs.

Retrieving Job Results

Users can query the status of their jobs using the /r/job/query call which returns the current status for a given job. Users can cancel a scheduled job or an actively running job using the /r/job/cancel call.

Regardless of whether a job runs to completion or whether a job is interrupted or aborted by the user the DeployR server will create a new project for the user by default. This project will be named after the job that created it and will contain a complete workspace, working directory, package dependencies, command history and associated results generated by that job. The user can then work with the full set of Project APIs to manipulate, analyze, visualize and interpret all of the objects, files and generated plots resulting from the job.

However, if the storenoproject parameter is set to true when submitting the job then a project is not created when the job completes. Using the storenoproject parameter can be very resource efficient and improve throughput when the only data you care about has already been stored in the repository using the storeobjects and/or storefiles parameters.

If the storenoproject parameter is not set to true then a project is always created to hold the results of a job unless the job experiences a runtime error during execution, for example, the R code being executed on the job has syntax errors or attempts to reference undefined objects. In such cases, see the statusMsg property on a failed job for an indication of the underlying cause of the problem.

Important! All completed, interrupted and aborted jobs result in new projects for users.

7.1 /r/job/list

This call lists jobs for the currently authenticated user.

The openonly parameter allows the caller to see only those jobs in an open state. The set of job open states are shown here:

  • Scheduled : job is scheduled but not yet queued for running.
  • Queued : job is queued for running.
  • Running : job is running.

Important! Any open job can be cancelled using the /r/job/cancel call.

HTTP Method

GET

Request Format

Content-Type: application/x-www-form-urlencoded; charset=utf-8

Parameters

  • format – specifies markup encoding on the call: json or xml
  • openonly – (optional) if true, only jobs in an open-state are listed in the response markup
  • extended - (optional) If true, additional data properties describing each job are listed in the response markup

7.2 /r/job/submit

This call submits a job for background execution on behalf of the user.

To submit the execution of an arbitrary block of R code the caller must provide a value on the code parameter.

To submit the execution of a single repository-managed script the caller must provide parameter values for rscriptname , rscriptauthor and optionally rscriptdirectory and rscriptversion . To submit the execution of a chain of repository-managed scripts the caller must provide a comma-separated list of values on the rscriptname , rscriptauthor and optionally rscriptdirectory and rscriptversion parameters.

To submit the execution of a single external script the caller must provide a valid URL or file path using the externalsource parameter. To submit the execution of a chain of external scripts the caller must provide a comma-separated list of values on the externalsource parameter. Note, to submit the execution of an external script the caller must have been granted the POWER_USER role.

Note: A chained execution executes each of the scripts identified on the call in a sequential fashion on the R session for the job, with execution occurring in the order specified on the parameter list.

Please see the Standard Execution Model section in this guide for further details about the parameters on this call.

HTTP Method

POST

Request Format

Content-Type: application/x-www-form-urlencoded; charset=utf-8

Parameters

  • format – specifies markup encoding on the call: json or xml
  • name – job name
  • descr – (optional) job description
  • code – (optional) R code to execute on job
  • rscriptname – (optional) comma-separated list of repository-managed script filenames to execute on job
  • rscriptdirectory - (optional) comma-separated list of repository-managed directories for scripts, defaults to root
  • rscriptauthor – (optional) comma-separated list of authors, one author per rscriptname
  • rscriptversion – (optional) comma-separated list of versions, one version per rscriptname
  • externalsource – (optional) comma-separated list of URLs or file paths to external scripts

Pre-Execution Parameters

  • inputs - (optional) DeployR-encoded inputs
  • csvinputs - (optional) comma-separated list of primitive name/value inputs
  • preloadbydirectory - (optional) comma-separated list of repository directory names
  • preloadfilename - (optional) comma-separated list of repository filenames
  • preloadfiledirectory - (optional) comma-separated list of directories, one directory per preloadfilename
  • preloadfileauthor - (optional) comma-separated list of authors, one author per preloadfilename
  • preloadfileversion - (optional) comma-separated list of versions, one version per preloadfilename
  • preloadobjectname - (optional) comma-separated list of repository object (.rData) filenames
  • preloadobjectdirectory - (optional) comma-separated list of directories, one directory per preloadobjectname
  • preloadobjectauthor - (optional) comma-separated list of authors, one author per preloadobjectname
  • preloadobjectversion - (optional) comma-separated list of versions, one version per preloadobjectname
  • adoptworkspace – _(optional) identifies project whose workspace is to be adopted
  • adoptdirectory – _(optional) identifies project whose directory is to be adopted
  • adoptpackages – _(optional) identifies project whose package dependencies are to be adopted

On-Execution Parameters

  • cluster – _(optional) identifies a grid node cluster for targeted execution
  • priority - (optional) specifies the scheduling priority for the job: low (default), medium or high
  • tag - (optional) specifies a tag that labels the execution
  • echooff - (optional) if true R commands will not appear in the console output saved on the project execution history for the job
  • consoleoff - (optional) if true console output is not saved on the project execution history for the job
  • artifactsoff - (optional) when enabled, artfiacts generated in the working directory are neither cached to the database nor reported in the response markup
  • enableConsoleEvents - (optional) if true console events are delivered on the event stream when the job executes
  • graphics - (optional) specifies preferred R graphics device for execution: png or svg
  • graphicswidth - (optional) specifies preferred width for R graphics device images
  • graphicsheight - (optional) specifies preferred height for R graphics device images

Post-Execution Parameters

  • storefile - (optional) comma-separated list of working directory filenames
  • storeobject - (optional) comma-separated list of workspace object names
  • storeworkspace - (optional) filename (.rData) where workspace contents will be saved in the repository
  • storedirectory - (optional) repository directory for stored files and objects after the execution completes, defaults to root
  • storenewversion - (optional) if true, ensures each file stored in repository results in new version being created if needed
  • storepublic - (optional) if true, publishes each file stored in the repository
  • storenoproject - (optional) if true, no project persistence following job execution

Examples 

format=json
name=Submitted Analysis
descr=Demonstrates job submission.
rscriptname=Histogram of Auto Sales
rscriptauthor=testuser

7.3 /r/job/schedule

This call schedules a job for background execution on behalf of the user.

The schedstart parameter identifies the start time for the job. This parameter value is specified as UTC in milliseconds. The schedrepeat parameter indicates the number of times the job is to be repeated, and if omitted the job is executed just once. The schedinterval parameter indicates the interval, measured in milliseconds, between repeat executions.

To schedule the execution of an arbitrary block of R code the caller must provide a value on the code parameter.

To schedule the execution of a single repository-managed script the caller must provide parameter values for rscriptname , rscriptauthor and optionally rscriptdirectory and rscriptversion . To schedule the execution of a chain of repository-managed scripts the caller must provide a comma-separated list of values on the rscriptname , rscriptauthor and optionally rscriptdirectory and rscriptversion parameters.

To schedule the execution of a single external script the caller must provide a valid URL or file path using the externalsource parameter. To schedule the execution of a chain of external scripts the caller must provide a comma-separated list of values on the externalsource parameter. Note, to schedule the execution of an external script the caller must have been granted the POWER_USER role.

Note: A chained execution executes each of the scripts identified on the call in a sequential fashion on the R session for the job, with execution occurring in the order specified on the parameter list.

Please see the Standard Execution Model section in this guide for further details about the parameters on this call.

7.6 /r/job/delete

This call deletes the specified job.

Only jobs in one of the success, cancelled or incompleted states can be deleted. The set of states are shown here:

  • Completed : job execution has run to successful completion.
  • Interrupted : job execution has been interrupted.
  • Cancelled : job has been cancelled.
  • Aborted : job execution has been aborted.
  • Failed : job execution has resulted in failure.

Jobs that are in an open-state can not be deleted.

Important! Deleting jobs will not delete the projects that resulted from those jobs.

HTTP Method

POST

Request Format

Content-Type: application/x-www-form-urlencoded; charset=utf-8

Parameters

  • format – specifies markup encoding on the call: json or xml
  • job - specifies a comma-separated list of job identifiers

Examples 

format=json
job=JOB-91baaca8-574d-4703-b225-97e64ec63e70

Response Markup (JSON)

{
    "deployr": {
        "response": {
            "success": true,
            "call": "/r/job/delete"
        }
    }
}

Response Markup (XML)

<deployr>
  <response>
    <success>true</success>
    <call>/r/job/delete</call>
  </response>
</deployr>
